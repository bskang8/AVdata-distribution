## 1. 핵심 배경: 왜 HNSW인가

### 문제: 고차원 벡터 유사도 검색의 비효율

이 프로젝트에는 299,180개 클립의 bge-m3 임베딩(1024-dim)이 있다. 쿼리가 들어올 때마다 가장 유사한 클립을 찾으려면 유사도 계산이 필요하다.

**Brute-force 방식의 비용**:

$$\text{연산 횟수} = N \times D = 299{,}180 \times 1024 \approx 3억 \text{ 번의 부동소수점 연산}$$

이를 매 쿼리마다 수행하면 수백 ms가 걸린다. 실시간 검색에는 부적합하다.

**HNSW의 해법**: 전체를 탐색하지 않고 **"탐색할 가치 있는 노드"만 추려낸 뒤** 거리를 계산한다.

$$\text{실제 계산 노드 수} \approx O(\log N) \text{ 수준}$$

| 방식 | 거리 계산 횟수 | 응답 시간 | Recall |
|------|-------------|---------|--------|
| Brute-force | 299,180번 | ~300ms | 100% |
| HNSW (이 프로젝트 설정) | ~수백~수천번 | **~5ms** | ~98% |

## 2. 기본 아이디어: NSW에서 HNSW로

### 2-1. NSW (Navigable Small World)

소셜 네트워크 이론의 "6단계 분리(six degrees of separation)"에서 착안한다. 각 노드가 **가까운 이웃(단거리 링크)**과 **멀리 있는 허브(장거리 링크)**를 동시에 유지하면, 어떤 시작점에서도 목적지까지 **로그 스케일 홉**으로 도달할 수 있다.

```
● ─────────────────────────────● (장거리 링크: 빠른 이동)
● ─── ● ─── ● ─── ● ─── ● ─── ● (단거리 링크: 정밀 탐색)
```

**문제점**: NSW 단일 계층에서는 삽입 순서에 따라 그래프 품질이 달라지고, 조기 수렴(local minimum) 문제가 발생한다.

### 2-2. HNSW (Hierarchical NSW)

NSW를 **여러 계층(레이어)**으로 쌓아 스킵리스트(Skip List) 구조를 만든다.

```
Layer 2:  ●─────────────────────────────●
          (노드 수 적음, 허브만 존재, 장거리 이동)

Layer 1:  ●──────●──────●──────●──────●
          (중간 밀도, 중거리 이동)

Layer 0:  ●─●─●─●─●─●─●─●─●─●─●─●─●─●
          (전체 노드, 단거리 정밀 탐색)
```

**핵심 통찰**:
- 상위 레이어 → 고속도로 (멀리 빠르게)
- 하위 레이어 → 골목길 (정밀하게)
- 탐색은 항상 **상위 → 하위**로 내려오며 점점 범위를 좁힌다

## 3. 인덱스 구성 알고리즘 (빌드 단계)

### 3-1. 노드 삽입 전체 흐름

벡터를 하나씩 삽입하면서 그래프를 구성한다. 논문 Algorithm 1 기준.

```
for 각 벡터 v in 데이터셋:
    1. 이 벡터가 존재할 최고 레이어 ℓ 결정 (확률적)
    2. 현재 진입점(entry point)에서 탐색 시작
    3. Layer L_max → Layer ℓ+1: greedy descent (ef=1, 빠르게 내려옴)
    4. Layer ℓ → Layer 0: 각 레이어에서 이웃 M개 선택
    5. 양방향 엣지 추가 (v↔이웃)
    6. 첫 번째 삽입된 노드면 진입점 업데이트
```

### 3-2. 레이어 배정: 확률적 계층 결정

각 노드의 최고 레이어 `ℓ`은 삽입 시 무작위로 결정된다.

$$\ell = \left\lfloor -\ln(\text{uniform}(0,1)) \cdot m_L \right\rfloor$$

$$m_L = \frac{1}{\ln M}$$

여기서 $M$은 노드당 최대 이웃 수 (이 프로젝트: M=32).

**결과 분포** (M=32 기준):

| 레이어 | 존재 확률 | 전체 299k 중 노드 수 (근사) |
|--------|---------|--------------------------|
| Layer 0 | 100% | 299,180개 |
| Layer 1 | ~28.8% | ~86,000개 |
| Layer 2 | ~8.3% | ~24,800개 |
| Layer 3 | ~2.4% | ~7,100개 |
| Layer 4+ | <1% | ~2,000개 미만 |

→ 자연스럽게 **역피라미드 형태**의 희소 계층이 형성된다.

### 3-3. 이웃 선택 휴리스틱 (Algorithm 4)

단순히 가장 가까운 M개를 고르지 않는다. **다양성 강제 휴리스틱**을 사용한다.

```
후보 집합 C에서 이웃 R을 선택할 때:
  새 후보 e를 R에 추가하려면:
    dist(e, v) < dist(e, r)  for any r ∈ R
  을 만족해야 함

→ 같은 방향의 중복 이웃 방지
→ 이웃이 여러 방향으로 분산되어 탐색 시 local minimum 탈출 용이
```

**직관적 설명**:

```
v를 중심으로 볼 때:

나쁜 이웃 선택 (모두 같은 방향):    좋은 이웃 선택 (다양한 방향):
       v                                    v
       │  × × ×                          / │ \
       └──────→ 이웃 3개 (동쪽만)       ↙  ↓  ↘
                탐색 시 서쪽 놓침      이웃들이 분산 → 어느 방향도 커버
```

**클러스터링 없이 만들어진 구조적 특성**

삽입이 끝나면 다음 구조가 자동으로 형성되어 있다.

```
"맑은 낮" 의미 클러스터: v₁ ─── v₂  (거리 0.31, 촘촘)
"야간 안개" 의미 클러스터: v₃ ─── v₄ ─── v₅  (0.21, 0.24, 촘촘)
클러스터 간 브릿지: v₂ ─── v₃  (거리 0.78, 느슨)
```

클러스터 경계를 명시적으로 정의하지 않았지만, **거리 기반 이웃 연결이 자동으로 의미 유사 노드끼리 밀집 연결**하고, 의미가 다른 노드 간에는 약한 연결만 남긴다.

### 3-5. 비클러스터링 방식의 장단점 및 도입 이유

#### 클러스터링 기반 방식과의 비교

| 항목 | 클러스터링 (k-means IVF) | HNSW 확률적 삽입 |
|------|------------------------|----------------|
| 사전 처리 | 전체 데이터 수집 후 k-means 실행 필요 | 없음 |
| 상위 계층 의미 | "이 클러스터의 중심점" | "우연히 허브가 된 노드" |
| 동적 추가 | 새 데이터 추가 시 클러스터 재계산 | 새 노드 삽입만으로 완료 |
| 공간 분할 | 보로노이 셀 경계 존재 | 경계 없음, 연속적 |
| 경계 근처 벡터 처리 | 잘못된 클러스터에 배정될 수 있음 (경계 효과) | 해당 없음 |
| 클러스터 수 k 지정 | 필수, 하이퍼파라미터 튜닝 필요 | 불필요 |
| 빌드 시간 복잡도 | $O(N \cdot k \cdot iter)$ | $O(N \cdot \log N)$ |
| 검색 시 후보군 | k개 클러스터 중 nprobe개만 탐색 | 그래프 경로를 따라 탐색 |

#### 비클러스터링 방식의 장점

**1. 동적 삽입 지원 (가장 중요한 실용적 이유)**

```
클러스터링(IVF): 새 데이터 1개 추가
  → 전체 k-means 재실행 또는 배치 단위 재빌드 필요
  → 실시간 업데이트 불가

HNSW: 새 데이터 1개 추가
  → insert(v_new) 한 번 호출
  → O(log N) 시간에 즉시 완료, 기존 그래프 유지
```

이 프로젝트처럼 클립이 지속적으로 추가되는 환경에서 결정적 우위를 가진다.

**2. 경계 효과(boundary effect) 없음**

클러스터링 기반 IVF는 클러스터 경계 근처의 벡터를 잘못된 클러스터에 배정할 수 있다. nprobe를 늘리면 완화되지만 속도가 희생된다. HNSW는 공간을 명시적으로 분할하지 않으므로 이 문제가 구조적으로 존재하지 않는다.

**3. 하이퍼파라미터가 단순**

IVF: k(클러스터 수), nprobe(탐색 클러스터 수), 재빌드 주기 등 복합 튜닝 필요  
HNSW: M, efConstruction, efSearch 3개로 정확도-속도 트레이드오프를 직관적으로 조절 가능

**4. 고차원에서 우수한 recall**

k-means 기반 클러스터링은 차원이 높아질수록 "모든 점이 거의 같은 거리" 문제(차원의 저주)로 클러스터 품질이 급격히 저하된다. HNSW는 탐색이 그래프 경로 기반이므로 고차원에서도 성능 저하가 적다. 1024-dim 임베딩에서 HNSW가 IVF보다 높은 recall을 달성하는 이유다.

#### 비클러스터링 방식의 단점

**1. 메모리 사용량**

```
IVF (클러스터 인덱스): 클러스터 중심 벡터만 메모리에
                       → 수백 MB로 수억 벡터 관리 가능
HNSW: 모든 벡터 + 그래프 구조를 메모리에 유지
      → 299k × 1024 × 4 bytes = 1.15 GB + 그래프 ~150 MB
      → 벡터 수가 수억 단위가 되면 RAM 한계 초과
```

이 프로젝트 규모(299k)에서는 문제없지만, 수억 개 이상의 벡터에서는 IVF+PQ 조합이 현실적이다.

**2. 빌드 시간**

```
HNSW: 삽입 시 각 벡터마다 greedy 탐색 반복
      → 299k 벡터 빌드 시간: 수십 분 (이 프로젝트 실제)
IVF:  k-means 수렴 후 배정 → 대량 데이터에서 더 빠를 수 있음
```

**3. 삽입 순서 민감성**

초기 삽입 벡터들이 진입점 후보가 되어 그래프 구조에 영향을 준다. 데이터 순서가 극단적으로 편향되면 그래프 품질이 저하될 수 있다. 일반적인 경우에는 무시할 수 있는 수준이다.

#### HNSW 설계 방식을 도입한 주요 이유

Malkov & Yashunin(2016)이 클러스터링을 의도적으로 배제하고 확률적 삽입을 선택한 데는 세 가지 핵심 근거가 있다.

**이유 1: 스킵리스트의 증명된 성질 활용**

스킵리스트(Pugh, 1990)는 동일한 확률적 계층 배정으로 $O(\log N)$ 탐색을 보장한다. HNSW는 스킵리스트의 1차원 아이디어를 고차원 벡터 공간으로 확장한 것이다. 확률적 계층 배정이 탐색 복잡도의 이론적 보장을 제공한다.

$$P(\ell \geq k) = e^{-k \cdot \ln M} = M^{-k}$$

이 분포가 각 레이어에 $N / M^k$개의 노드를 배치해 $O(\log_M N)$ 레이어를 만든다.

**이유 2: 점진적 구축의 실용성**

클러스터링은 "전체를 보고 구조를 결정"하는 방식이지만, 현실에서 데이터는 스트리밍으로 도착한다. 확률적 삽입은 각 데이터를 독립적으로 처리하므로 배치 수집 없이 실시간으로 인덱스를 유지할 수 있다.

**이유 3: 이웃 선택 휴리스틱이 클러스터링의 역할을 대체**

클러스터링이 하는 핵심 작업은 "같은 공간의 벡터들을 같은 그룹에 넣는 것"이다. HNSW의 다양성 강제 이웃 선택 휴리스틱(Algorithm 4)이 이를 암묵적으로 수행한다. 가까운 벡터들이 서로 이웃으로 연결되면, 탐색 시 그래프 경로가 자동으로 의미적으로 유사한 영역으로 수렴한다. 명시적 클러스터 경계 없이 동일한 효과를 달성한다.

## 4. 파일 저장 구조

### 4-1. IndexHNSWFlat의 저장 내용

`faiss.write_index()`가 직렬화하는 내용:

```
hnsw.index (약 1.3 GB)
├── 헤더: 인덱스 타입, 파라미터 (dim, M, efConstruction, efSearch)
├── 벡터 스토리지 (FlatStorage)
│   └── 299,180 × 1024 × 4 bytes (float32) ≈ 1.15 GB
└── HNSW 그래프
    ├── 각 노드의 레이어별 이웃 목록 (neighbor lists)
    │   Layer 0: 각 노드당 최대 2M=64개 이웃
    │   Layer 1+: 각 노드당 최대 M=32개 이웃
    ├── 각 노드의 최고 레이어 정보 (ℓ 값)
    └── 현재 진입점 (entry point) 노드 ID
```

**왜 Flat인가?**

`IndexHNSWFlat`은 벡터를 압축하지 않고 원본 float32로 저장한다.

| 인덱스 타입 | 벡터 저장 | 크기 | 정확도 |
|-----------|---------|------|--------|
| `IndexHNSWFlat` (이 프로젝트) | 원본 float32 | ~1.3 GB | 최대 |
| `IndexHNSWPQ` | Product Quantization (압축) | ~200 MB | 약간 감소 |
| `IndexHNSWSQ` | Scalar Quantization | ~350 MB | 소폭 감소 |

이 프로젝트는 정확도를 최대로 유지하기 위해 Flat을 선택했다.

### 4-2. 파일 크기 산출

$$\text{벡터 스토리지} = 299{,}180 \times 1024 \times 4 \approx 1{,}151 \text{ MB}$$

$$\text{그래프 오버헤드} \approx \text{평균 이웃 수} \times \text{노드 수} \times 4 \text{ bytes} \approx 150\text{~}200 \text{ MB}$$

$$\text{합계} \approx 1.3 \text{ GB}$$

## 5. 검색 단계별 상세 분석

쿼리 텍스트 "야간 안개 고속도로에서 트럭이 급제동하는 상황"이 들어왔을 때의 전체 흐름.

### Step 1: 쿼리 임베딩

```
쿼리 텍스트: "야간 안개 고속도로에서 트럭이 급제동하는 상황"
    │
    ▼  bge-m3 인코더 (SentenceTransformer)
쿼리 벡터 q: [0.023, -0.147, 0.381, ..., -0.092]  ← 1024차원 float32
```

### Step 2: 최상위 레이어 진입

```
현재 진입점: entry_point_node (빌드 시 마지막으로 설정된 허브 노드)

Layer L_max에서 시작:
  현재 노드 c = entry_point_node
  repeat:
    c의 이웃들 중 q에 더 가까운 노드가 있으면 → 이동
    없으면 → 이 레이어에서 local minimum 도달
  → Layer L_max-1로 하강, 현재 c를 진입점으로 사용
```

이 과정을 **ef=1** (단 하나의 후보만 유지)로 Layer 1까지 반복한다. 정밀도보다 속도 우선.

### Step 3: Layer 0 정밀 탐색

Layer 0에 도달하면 **efSearch=128** 크기의 후보 집합을 유지하며 탐색한다.

```
후보 집합 W = {진입점 c}  (우선순위 큐, 거리 기준 정렬)
방문 집합 V = {c}

while W가 비지 않음:
    c_nearest = W에서 q와 가장 가까운 노드 꺼냄
    
    if dist(c_nearest, q) > dist(W의 가장 먼 노드, q):
        break  ← 더 가까운 것을 찾을 수 없음
    
    for c_nearest의 각 이웃 e:
        if e ∉ V:
            V에 e 추가
            if dist(e, q) < dist(W의 가장 먼 노드, q) or |W| < efSearch:
                W에 e 추가
                if |W| > efSearch:
                    W에서 가장 먼 노드 제거  ← 크기 efSearch 유지

return W의 상위 top-k
```

**efSearch=128의 의미**: Layer 0 탐색 중 항상 최대 128개의 후보를 유지한다. 실제로 거리를 계산하는 노드 수는 수백~수천 개 수준이다.

### Step 4: 결과 반환 및 clip_ids 매핑

```
FAISS 반환: [(거리 0.142, 인덱스 84231), (거리 0.156, 인덱스 12045), ...]
    │
    ▼  clip_ids.json[인덱스]
최종 반환: ["clip_084231", "clip_012045", ...]  ← top-k 클립 ID
```

### 전체 흐름 요약

```
쿼리 텍스트
    │
    ▼ [Step 1] bge-m3 인코딩 (~100ms, 최초 1회 모델 로드 후 ~5ms)
쿼리 벡터 (1024-dim)
    │
    ▼ [Step 2] 상위 레이어 greedy descent (ef=1)
   Layer L_max ──greedy──→ Layer 2 ──greedy──→ Layer 1
   (각 레이어: 수십 번 이동, 빠른 위치 추정)
    │
    ▼ [Step 3] Layer 0 정밀 탐색 (ef=efSearch=128)
   후보 집합 W (최대 128개) 유지하며 BFS 유사 탐색
   실제 거리 계산: 수백~수천 번 (전체 299k 대비 <1%)
    │
    ▼ [Step 4] top-k 추출 + clip_ids 매핑
결과 클립 ID 리스트
```

## 6. 거리 메트릭: L2 거리와 코사인 유사도의 동치성

HNSW 그래프를 구성하고 탐색하려면 "가깝다"를 정의하는 메트릭이 반드시 필요하다. 이 프로젝트에서 FAISS가 내부적으로 사용하는 메트릭과 그것이 의미론적으로 무엇을 측정하는지 정확히 확인한다.

### 6-1. 코드에서 확인된 두 가지 핵심 결정

**결정 1: 인덱스 생성 시 메트릭 — L2 (유클리드 제곱 거리)**

```python
# src/avdata/phase2/build_embeddings.py
index = faiss.IndexHNSWFlat(dim, HNSW_M)
# 메트릭 미지정 → FAISS 기본값 = METRIC_L2 (유클리드 제곱 거리)
```

`IndexHNSWFlat`의 두 번째 인자는 M(이웃 수)이고, 메트릭을 명시하지 않으면 `faiss.METRIC_L2`가 기본값으로 적용된다.

**결정 2: 임베딩 시 벡터 정규화**

```python
# src/avdata/phase2/build_embeddings.py (빌드 시)
embeddings_sorted = model.encode(..., normalize_embeddings=True, ...)

# src/avdata/search/searcher.py (검색 시)
q_emb = self._emb_model.encode([query], normalize_embeddings=True)
```

빌드와 검색 양쪽에서 일관되게 `normalize_embeddings=True`를 적용한다.

### 6-2. L2 거리가 코사인 유사도와 같아지는 수학적 증명

L2 제곱 거리의 정의:

$$d_{L2}^2(\mathbf{u}, \mathbf{v}) = \|\mathbf{u} - \mathbf{v}\|^2 = \|\mathbf{u}\|^2 + \|\mathbf{v}\|^2 - 2\,\mathbf{u} \cdot \mathbf{v}$$

`normalize_embeddings=True`로 인해 모든 벡터는 단위 구면에 투영된다.

$$\|\mathbf{u}\| = 1, \quad \|\mathbf{v}\| = 1$$

이를 대입하면:

$$d_{L2}^2(\mathbf{u}, \mathbf{v}) = 1 + 1 - 2\,\mathbf{u} \cdot \mathbf{v} = 2 - 2\cos\theta$$

따라서:

$$d_{L2}^2 \text{ 최소화} \iff \cos\theta \text{ 최대화} \iff \text{코사인 유사도 최대화}$$

**결론**: FAISS는 L2 거리를 계산하지만, 벡터가 모두 정규화되어 있으므로 **실질적으로 코사인 유사도 기반 의미 검색**이다.

### 6-3. 단위 구면 위의 기하학적 직관

```
            단위 구면 단면도 (1024-dim → 2-dim 투영)

                    ● v₁  ("안개 야간 클립" 임베딩)
                   /  ↑ 각도 θ₁ 작음 = 코사인 유사도 높음
        q ●───────   ← 쿼리: "야간 안개 고속도로"
                   \  ↓ 각도 θ₂ 큼 = 코사인 유사도 낮음
                    ● v₂  ("맑은 낮 클립" 임베딩)

    모든 벡터가 반지름 1인 구 위에 있으므로
    "가깝다" = "같은 방향을 가리킨다" = "의미적으로 유사하다"
```

두 벡터가 동일한 방향이면 $\cos\theta = 1$, $d_{L2}^2 = 0$ (완전 일치).  
두 벡터가 직교하면 $\cos\theta = 0$, $d_{L2}^2 = 2$.  
두 벡터가 반대 방향이면 $\cos\theta = -1$, $d_{L2}^2 = 4$ (최대 거리).

### 6-4. FAISS에서 내적을 직접 메트릭으로 쓰는 방법

정규화 없이 내적을 직접 사용하고 싶다면:

```python
# METRIC_INNER_PRODUCT 명시
index = faiss.IndexHNSWFlat(dim, M, faiss.METRIC_INNER_PRODUCT)
```

그러나 이 경우 벡터 크기(magnitude)가 유사도에 영향을 주므로, **정규화 + L2가 표준**이다. bge-m3 공식 문서도 `normalize_embeddings=True`를 권장한다.

### 6-5. 메트릭이 HNSW 그래프 품질에 미치는 영향

HNSW 그래프는 삽입 시 "이 벡터와 가장 가까운 이웃을 찾아 연결"하는 방식으로 구성된다. 따라서 **메트릭이 곧 그래프의 이웃 관계를 정의**한다.

- L2 + 정규화(= 코사인): 의미적으로 유사한 캡션끼리 이웃으로 연결
- 결과: "안개", "야간", "고속도로" 관련 클립들이 임베딩 공간에서 밀집된 클러스터를 형성하고, HNSW 그래프에서도 이 클러스터 내 노드들이 서로 이웃으로 연결됨
- 탐색 시: 쿼리 "야간 안개 고속도로"가 해당 클러스터 방향으로 greedy하게 이동 → 효율적으로 관련 클립 도달

| 파라미터 | 값 증가 시 | 값 감소 시 |
|---------|---------|---------|

| `efSearch` | recall ↑, 검색 느림 | 검색 빠름, recall ↓ |

이 프로젝트는 세 파라미터 모두 **보수적(품질 우선)** 으로 설정했다.
